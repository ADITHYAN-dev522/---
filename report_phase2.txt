CHAPTER 1
INTRODUCTION

Cybersecurity in the digital era demands more than isolated detection tools. As organizations expand their digital footprints across cloud platforms, containerized workloads, and remote endpoints, the attack surface grows proportionally. Traditional security operations rely on siloed tools such as standalone antivirus software, periodic vulnerability scanners, and manual incident review processes. These approaches generate excessive alert noise, fail to correlate related events, and provide no structured memory of past incidents.

Phase 1 of the SentinelNexus project established the foundational architecture for an integrated cybersecurity platform, implementing the initial vulnerability scanning pipeline using OSV Scanner, Trivy, and Semgrep, a host telemetry collection system, a basic ClamAV-based malware detection service, and a React-based dashboard with an Ollama-powered AI assistant.

Phase 2 significantly extends the platform by introducing four critical additions that were proposed but not yet implemented in Phase 1: a real YARA-based static analysis engine for file-level threat detection, a VirusTotal API integration for external hash reputation lookup, a SQLite-backed incident memory store for persistent threat tracking, and a ThreatSentinel correlation engine that aggregates findings across all scanners into unified, actionable threat events. Additionally, Phase 2 adds a multi-factor Risk Scoring Engine and a PatchMaster remediation advisor that maps detected CVEs to executable fix commands.

Together, these additions fulfill the complete proposed architecture described in Phase 1 with fully verifiable, deployed code. The platform is now capable of performing end-to-end threat detection, multi-source correlation, risk quantification, and guided remediation — all within a single unified interface.


CHAPTER 2
LITERATURE SURVEY

2.1 YARA Rules in Malware Detection and Threat Intelligence (2024)
Authors: El-Bolock, A., Bousselham, R., Elgazzar, K.
This paper examines the practical effectiveness of YARA rules as a lightweight static analysis mechanism for identifying malware families. YARA (Yet Another Ridiculous Acronym) allows security researchers to define pattern-matching rules based on binary strings, regular expressions, and metadata conditions. The authors demonstrate that YARA rule scanning, when combined with hash-based reputation lookup, achieves significantly lower false-positive rates than antivirus-only approaches. They also highlight that community-maintained YARA rule repositories such as YARA-Rules and MalwareBazaar provide sufficient coverage for common malware families without requiring proprietary signatures. The paper recommends hybrid pipelines that combine static YARA scanning with dynamic behavioral analysis for comprehensive coverage.
Limitations: YARA rules are only as effective as their authorship — unmaintained or generic rules produce high false-positive rates. The paper also notes that heavily obfuscated malware can evade string-based YARA patterns entirely.

2.2 VirusTotal Integration for Enterprise Threat Intelligence Pipelines (2024)
Authors: Garcia, F., Hernandez, J., Uribe, C.
This paper proposes architectural patterns for integrating VirusTotal's public API into automated threat intelligence workflows. The authors discuss rate-limiting constraints of the free tier (four requests per minute), caching strategies to avoid redundant lookups, and the value of multi-engine consensus (averaging detection rates across 70+ antivirus engines) for reducing false positives. The paper demonstrates that hash-first lookup (SHA-256 comparison against VT's index) is significantly faster than file submission while covering the majority of known threats. They also analyze the metadata available per file, including malware family classification and first-seen dates, which enables timeline-based threat analysis.
Limitations: VirusTotal's free tier has strict rate limits. Unknown or newly compiled malware will not appear in the database, limiting zero-day coverage.

2.3 Incident Correlation in Modern Security Operations Centers (2025)
Authors: Balogun, T., Schwartz, A., Lee, J.
This systematic review examines how correlation engines in modern SOC platforms reduce alert fatigue. The authors analyze twenty-three correlation approaches including rule-based, graph-based, and machine learning driven methods. Key findings show that even simple signature-based correlation that groups alerts by host, time window, and attack technique reduces analyst review time by 42% compared to reviewing individual alerts. The paper recommends that correlation engines maintain a persistent memory store to enable pattern recognition across multiple scan cycles, not just within a single run.
Limitations: Correlation engines trained on specific datasets may miss novel attack chains. The paper acknowledges that ground-truth labeled datasets for correlation quality evaluation remain scarce.

2.4 Risk Scoring Frameworks for Vulnerability Prioritization (2025)
Authors: Okonkwo, E., Fischer, R., Patel, S.
This paper surveys risk scoring methodologies used in enterprise vulnerability management programs. The authors compare CVSS-only scoring against multi-factor approaches that incorporate asset criticality, exploitability evidence (such as VirusTotal detections), and environmental context. Their findings show that pure CVSS scoring leads to high-severity vulnerability fatigue because many CRITICAL-scored CVEs have no public exploit or low real-world exposure. Multi-factor scoring that weights actual exploitation evidence (presence in malware databases, active network exposure) produces a more actionable priority ordering. The paper proposes a scoring model combining static severity, runtime evidence, and asset importance — a model closely aligned with the Risk Scoring Engine implemented in Phase 2 of SentinelNexus.
Limitations: Multi-factor scores require continuous data feeds and may be difficult to reproduce consistently across scan cycles.

2.5 Automated Remediation Guidance Using CVE-to-Package Mappings (2024)
Authors: Zhao, W., Kumar, A., Thompson, D.
This paper investigates the feasibility of automatically generating remediation commands from vulnerability scan outputs. The authors analyze popular package ecosystems (pip, npm, apt, gem, cargo) and demonstrate that CVE-to-package-manager mappings can be inferred from Trivy and OSV scanner outputs with high accuracy. The paper proposes a two-stage pipeline: first, normalize scanner outputs to extract affected package name, version, and ecosystem; second, generate the appropriate upgrade command for the detected package manager. The authors report that automated command generation reduces mean time to remediation in controlled experiments by reducing the manual lookup step that analysts typically perform.
Limitations: Auto-generated commands may fail for packages with complex dependency graphs or those requiring manual configuration changes after upgrade.


CHAPTER 3
EXISTING SYSTEM

3.1 Existing System
Traditional cybersecurity environments rely on fragmented, independently operated tools. Security Information and Event Management (SIEM) systems aggregate logs but produce overwhelming alert volumes without structured correlation. Point-in-time vulnerability scanners report findings but do not track remediation history or recurring exposure. Antivirus tools such as ClamAV provide signature-based detection but lack static analysis depth and external reputation validation. Threat intelligence platforms provide indicator feeds but require manual analyst correlation to connect them to system-specific findings.

3.1.1 Operations of the Existing System
1. Independent Tool Operation: Each scanner (antivirus, vulnerability scanner, SIEM) runs in isolation. Results are stored in separate formats and locations, requiring manual effort to understand compound risk.
2. No Persistent Incident Memory: Traditional systems treat each scan cycle as independent. A vulnerability detected in one run and left unpatched generates a new alert in the next run with no historical context about its age or prior investigation.
3. Manual Alert Correlation: Analysts must manually determine whether a ClamAV detection and a high-severity CVE in the same system are related. Without structured correlation, the connection is often missed.
4. Static Recommendations: Responses consist of generic advice. No tool automatically maps a specific CVE to the exact package manager command needed to patch it.
5. No External Hash Validation: Standard ClamAV scanning does not query external intelligence databases. A file flagged by YARA rules cannot be automatically cross-referenced against the VirusTotal multi-engine consensus.

3.1.2 Limitations of the Existing System
1. Alert Fatigue: Unconnected per-tool alerts overwhelm analysts. Critical patterns hidden across tools go unnoticed.
2. No Memory of Past Incidents: Systems start fresh each cycle, missing recurring attacks and unresolved exposures.
3. Weak Malware Analysis Depth: ClamAV alone cannot detect files with obfuscated or novel payloads that match known behavioral patterns (YARA rules) but have modified signatures.
4. No Risk Quantification: There is no unified score telling operators which combination of findings represents the highest current risk.
5. Manual Remediation Lookup: Analysts must manually search for the correct patch command for each CVE, adding minutes per finding to total remediation time.


CHAPTER 4
PROPOSED SYSTEM

The Phase 2 SentinelNexus platform extends the Phase 1 foundation by integrating the five major modules that were missing from the original deployed codebase.

4.1 Objectives of Phase 2

1. Real Static Malware Analysis (YARA Engine)
Deploy a functioning YARA-based static analysis engine that scans files for known malicious patterns using community-maintained rule sets. The engine also produces MD5, SHA-1, and SHA-256 multi-hash fingerprints for each scanned file, enabling external intelligence lookups.

2. External Hash Reputation Validation (VirusTotal Integration)
For every file flagged by ClamAV or YARA, submit its SHA-256 hash to the VirusTotal public API v3. Return the multi-engine detection count, malware family classification, and threat label. If VT_API_KEY is not configured, the pipeline gracefully skips this step without breaking the rest of the system.

3. Persistent Incident Memory (SQLite Store)
Maintain a structured database of all detected incidents, classified by severity, scanner source, and status (open or resolved). Track recurring attack patterns separately so repeated exposures are immediately visible. The memory store enables the system to learn across scan cycles rather than treating each run independently.

4. Threat Correlation Engine (ThreatSentinel)
After each combined scan cycle, aggregate findings from ClamAV, YARA, VirusTotal, Trivy, and Wazuh into unified ThreatEvent objects. Sort events by severity and persist them to a cache file accessible to the frontend in real time.

5. Multi-Factor Risk Scoring (Risk Scoring Engine)
Compute a 0-100 platform risk score by combining: ClamAV infection count, YARA rule hit count, VirusTotal detection positives, critical and high vulnerability count from Trivy, and Wazuh alert severity. Return a human-readable risk label (LOW/MEDIUM/HIGH/CRITICAL) with a breakdown of each contributing factor.

6. CVE-to-Remediation Mapping (PatchMaster)
Parse Trivy and OSV scan outputs to extract affected package names, installed versions, fixed versions, and package ecosystems. Generate the exact package manager command required to patch each vulnerability (pip install --upgrade, apt-get install --only-upgrade, npm install, etc.) with a direct link to the CVE advisory.

4.2 System Architecture of Phase 2

The Phase 2 architecture adds three new layers to the Phase 1 platform:

Layer 1 — Detection Layer (Extended):
ClamAV (signature-based antivirus) now operates in parallel with the YARA Engine (pattern-based static analysis) and VirusTotal API (external reputation lookup). These three detection mechanisms together cover known signatures, behavioral patterns, and global intelligence consensus.

Layer 2 — Memory and Correlation Layer (New):
The SQLite Memory Module stores all incidents across scan cycles and tracks recurring patterns. The ThreatSentinel Correlator reads from all scanner outputs after each cycle and produces a unified threat event list sorted by severity. The Risk Scoring Engine reads from all sources and computes the platform risk score.

Layer 3 — Remediation Layer (New):
PatchMaster reads Trivy and OSV outputs and generates actionable fix commands, displayed to analysts in the Response page with one-click copy functionality.

CHAPTER 5
DESIGN AND IMPLEMENTATION

5.1 Design

Phase 2 preserves the three-layer architecture established in Phase 1 and adds the Memory/Correlation layer between the Detection Layer and the Decision Layer.

5.1.1 Design Goals
- Complete the proposed malware analysis pipeline: ClamAV alone detects known signature matches. YARA adds behavioral pattern recognition. VirusTotal adds global intelligence consensus. Together they form a credible three-stage malware analysis pipeline.
- Build reliable cross-cycle memory: The SQLite memory store persists incident records across server restarts and scan cycles, enabling the system to recognize that a vulnerability reported today was also present three days ago.
- Provide honest risk quantification: The Risk Scoring Engine produces a single actionable score based on what is actually detected, not a static placeholder value.
- Enable analyst-executable remediation: PatchMaster generates commands that can be copied directly into a terminal, reducing the lookup step for each CVE from minutes to seconds.
- Eliminate all hardcoded frontend data: Every dashboard element now reflects a live backend API call. No static mock data appears in any production page.

5.2 Implementation

5.2.1 YARA Engine (backend/app/services/yara_engine/scanner.py)
The YARA engine is implemented using the yara-python library. At module load time, a curated inline rule set is compiled from a multiline string containing six categories of rules: EICAR test file detection, obfuscated PowerShell invocation, bash reverse shell patterns, hardcoded credential strings, large base64 blobs (typical of encoded shellcode), and ELF/PE magic bytes embedded in non-binary files. Each file is first read into memory, then hashed using hashlib for MD5, SHA-1, and SHA-256. The compiled YARA rules are applied with a 30-second timeout. Matches are returned as structured dictionaries containing the rule name, namespace, tags, metadata (including severity), and string match offsets. Files larger than 10 MB are skipped to prevent memory pressure. If yara-python is not installed, the engine returns an informative error per file rather than crashing the server.

5.2.2 VirusTotal Integration (backend/app/services/virustotal/lookup.py)
The VirusTotal module calls the v3 files endpoint using the file's SHA-256 hash. The API key is read from the VT_API_KEY environment variable. HTTP 404 responses indicate an unknown hash (no prior submission); HTTP 429 indicates rate limiting. The free tier allows four requests per minute, so the batch lookup function inserts a 15-second delay between requests. The response is normalized to extract: total engine count, positive detection count, malware family (from popular_threat_classification), threat label (from meaningful_name), and up to 20 engine names that reported a detection, all returned in a consistent dictionary regardless of whether the VT key is configured.

5.2.3 SQLite Memory Module (backend/app/services/memory/db.py)
The memory module creates a SQLite database at backend/scans/memory.db. Two tables are used: incidents (id, timestamp, type, severity, scanner, title, details_json, status, resolved_at) and patterns (signature, first_seen, last_seen, hit_count, severity). Indices are created on status, severity, and timestamp for query performance. The add_incident function generates an 8-character UUID-derived ID and inserts the record. The update_pattern function upserts pattern records and increments hit_count on repeat encounters. The resolve_incident function updates status to resolved and records the resolution timestamp. The get_incidents function supports optional status and severity filters with a configurable limit parameter.

5.2.4 ThreatSentinel Correlator (backend/app/services/threat_sentinel/correlator.py)
The correlator reads from three sources: malware_status.json (ClamAV detections and YARA results), latest.json (Trivy vulnerability results), and wazuh_alerts.json (Wazuh alert cache, if present). From each source, it extracts normalized ThreatEvent dictionaries with consistent fields: id, timestamp, type, scanner, severity, title, details, and status. Events with critical or high severity from vulnerability scans are persisted to the SQLite memory module via add_incident. YARA matches and ClamAV detections are always persisted. The resulting list is sorted by severity then timestamp, written to threat_events.json, and returned. The get_events function returns the cached list or triggers a fresh correlation run if the cache does not exist.

5.2.5 Risk Scoring Engine (backend/app/services/risk_engine/scorer.py)
The scorer aggregates signals from four JSON files: malware_status.json, yara_results.json, vt_results.json, and latest.json. Each source contributes a capped subscore: ClamAV (max 35 points), YARA (max 25 points), VirusTotal (max 20 points), Trivy critical/high counts (max 15 points), and Wazuh high-severity alerts (max 5 points). The total is capped at 100 and mapped to four labels: LOW (0-24), MEDIUM (25-49), HIGH (50-74), CRITICAL (75-100). The response includes the full breakdown and a plain-language recommendation.

5.2.6 PatchMaster Advisor (backend/app/services/patchmaster/advisor.py)
The advisor parses latest.json to extract Trivy vulnerability entries and OSV results. For each unique CVE, it extracts the package name, installed version, fixed version, and target type (e.g., pip, apt). Package manager detection uses a heuristic function that matches the Trivy type string against known patterns (e.g., "python-pkg" maps to pip, "dpkg" maps to apt). The fix_command function then generates the appropriate install/upgrade command for the detected package manager. Results are sorted by severity (critical first) and paginated via a limit parameter.

5.2.7 Updated Malware Pipeline (backend/auto_scanner.py)
The malware scan pipeline now runs in four sequential stages within run_malware_scan(): Stage 1 runs run_clamav_scan() using clamscan. Stage 2 runs run_auditd_scan() using ausearch for process execution auditing. Stage 3 calls yara_scanner.scan_directory() on all configured scan paths and writes results to yara_results.json. Stage 4 extracts unique SHA-256 hashes from YARA-flagged files and calls vt_lookup.lookup_hashes(), writing results to vt_results.json. The risk score and verdict are recalculated incorporating yara_hits and vt_positives. A correlation run is triggered after each full malware scan cycle.

5.2.8 Frontend Pages (Real Data)
All five primary pages now fetch exclusively from backend APIs:
- Dashboard: Fetches /api/threat-sentinel/events and /api/risk/score. Displays actual detected threat events as cards and shows the platform risk score as a badge.
- Detections: Fetches /api/threat-sentinel/events with 60-second auto-refresh and a live search/severity filter.
- Incidents: Fetches /api/memory/incidents and /api/memory/stats. Displays open/resolved counts from the SQLite store and provides a Resolve button calling PATCH /api/memory/incidents/{id}/resolve.
- Response: Fetches /api/patchmaster/recommendations. Displays CVE-mapped fix commands with one-click copy-to-clipboard.
- Malware: Fetches /api/malware/status (ClamAV), /api/malware/yara-results (YARA matches with hashes), and /api/malware/vt-results (VirusTotal engine detections).


CHAPTER 6
RESULT

6.1 Overview

The Phase 2 implementation of SentinelNexus successfully deploys all five missing components identified in the Phase 1 audit, establishing a fully functional and verifiable integrated security platform. The system now operates a complete malware analysis pipeline (ClamAV signature detection, YARA static rule matching, VirusTotal external reputation lookup), a persistent SQLite incident memory store, a multi-source ThreatSentinel correlation engine, a multi-factor Risk Scoring Engine, and a CVE-to-command PatchMaster remediation advisor.

6.2 Implemented Capabilities

1. YARA Static Analysis Engine
The YARA engine successfully scans configured directories and applies six categories of detection rules. For each file, it computes MD5, SHA-1, and SHA-256 hashes. Rule matches are returned with rule name, detected string offsets, and metadata severity. The engine gracefully handles the case where yara-python is not installed, returning informative error messages rather than crashing. Files with confirmed YARA rule matches produce entries in yara_results.json and are surfaced in the Malware Analysis dashboard page under the YARA Static Analysis section.

2. VirusTotal Hash Reputation Lookup
The VirusTotal integration calls the v3 /files/{sha256} endpoint for each file flagged by YARA or ClamAV. The lookup returns the total engine count, positive detection count, malware family label, and names of specific engines that flagged the file. Results are written to vt_results.json. The Malware page displays a dedicated VirusTotal Hash Reputation section showing per-hash results with a direct link to the full VirusTotal report. When VT_API_KEY is not configured, the section informs the analyst rather than displaying an empty panel.

3. SQLite Incident Memory Store
The memory module persists all critical and high severity findings to the incidents table of memory.db across scan cycles. The /api/memory/stats endpoint returns real open, resolved, and total incident counts. The Incidents page displays these counts as real numbers, not hardcoded values. The resolve button on each incident calls the PATCH endpoint, marking the incident resolved with a timestamp. Recurring attack patterns are tracked in the patterns table, and the top five most-seen patterns are displayed in the Incidents page.

4. ThreatSentinel Correlation Engine
The correlation engine aggregates findings from ClamAV, YARA, and Trivy after each scan cycle. Events are sorted by severity (critical first) and cached in threat_events.json. The Dashboard page replaces the previously hardcoded DDoS/PrivEsc/DataExfil placeholder cards with real ThreatEvent cards fetched from /api/threat-sentinel/events. The Detections page provides a searchable, filterable list of all correlated events with 60-second auto-refresh.

5. Risk Scoring Engine
The risk scorer produces a 0-100 score combining ClamAV, YARA, VirusTotal, and Trivy signals. This score appears in the Dashboard as a real-time badge showing the label and numeric value. The color of the badge reflects the risk level (green for LOW, yellow for MEDIUM, orange for HIGH, red for CRITICAL). A breakdown of each contributing factor score is included in the API response.

6. PatchMaster Remediation Advisor
PatchMaster parses Trivy and OSV scan results and generates executable fix commands for each CVE. For a pip package, it generates pip install --upgrade {pkg}=={fixed_version}. For an apt package, it generates sudo apt-get install --only-upgrade {pkg}. The Response page lists all recommendations sorted by severity with a Copy button for each command and a link to the CVE advisory. The page also shows counts of total recommendations, auto-patchable items, and critical-priority items.

6.3 Frontend Verification

All six primary pages (Dashboard, Detections, Incidents, Response, Malware, Intelligence) fetch exclusively from live backend API routes. No hardcoded placeholder data remains in any production page. The backend status indicator in the Dashboard reflects actual connectivity state (green: connected, red: offline). The Malware page accurately reports ClamAV, YARA, and VirusTotal findings from the most recent scan cycle. The Incidents page resolves incidents in real time by calling the PATCH endpoint, and the open/resolved counts update after each resolve action.


CHAPTER 7
CONCLUSION AND FUTURE SCOPE

7.1 Conclusion

Phase 2 of SentinelNexus successfully implements every capability proposed but not yet deployed in Phase 1. The platform now delivers a three-stage malware analysis pipeline (ClamAV, YARA, VirusTotal), a persistent SQLite incident memory store with pattern tracking, a multi-source ThreatSentinel correlation engine, a multi-factor Risk Scoring Engine, and a CVE-to-command PatchMaster remediation advisor. All frontend pages display exclusively real data from live backend API endpoints, eliminating the hardcoded placeholder values present in Phase 1.

The AI Copilot continues to operate via Ollama and now receives real scan context (malware verdict, YARA hit count, VirusTotal positives, open incident list, critical vulnerability count) in every prompt, enabling factually grounded responses about the current system security posture.

The result is a platform that genuinely fulfills the integrated security vision described in Phase 1's proposed system chapter. Analysts can now: detect malware through three complementary mechanisms, view a real-time risk score, investigate correlated threat events, track incident resolution history, and execute copy-ready patch commands — all within a single unified interface.

7.2 Future Scope

1. Cuckoo Sandbox Integration
Dynamic malware analysis using a sandboxed execution environment would add a fourth layer to the malware pipeline. Files would be executed in an isolated container monitored by Sysmon or eBPF, with behavioral traces (file writes, network connections, registry changes) correlated against MITRE ATT&CK techniques.

2. FAISS Semantic Search Over Incident History
Adding a FAISS vector index on top of the SQLite incident store would enable semantic similarity search across past incidents. Analysts could query "find incidents similar to this ClamAV detection" and receive related historical events ranked by embedding similarity.

3. Neo4j Graph Database for Relationship Mapping
Replacing the flat JSON correlation cache with a Neo4j graph database would enable true relationship mapping between hosts, IPs, CVEs, malware families, and attack techniques. Graph traversal queries would expose multi-hop attack paths invisible to flat data structures.

4. MITRE ATT&CK Technique Mapping
Mapping YARA rule hits and ClamAV detections to the MITRE ATT&CK framework tactic and technique taxonomy would enable structured threat actor profiling and detection coverage gap analysis.

5. Automated Patch Execution (PatchMaster Active Mode)
Currently PatchMaster generates commands for manual execution. A governed active mode would allow approved commands to execute automatically on non-production assets, reducing time-to-remediation for routine patching tasks.

6. Multilingual Copilot Support
The Blue Team Copilot currently operates in English. Extending the system prompt and LLM context to support Arabic, French, Spanish, and other languages would make the platform accessible to global security operations teams.

7. Red Team Simulation Module
A dynamic red team simulator would execute controlled attack scenarios (phishing simulations, privilege escalation attempts, lateral movement probes) against the monitored environment to continuously validate that detection rules and YARA patterns are catching real adversarial behavior.

8. Continuous Learning via Feedback Loop
Analyst resolve actions and override decisions should be fed back into the risk scoring weights and YARA rule confidence thresholds. Over multiple cycles, the system would adapt its scoring to reflect which alerts have historically proven to be genuine threats versus false positives in the specific deployment environment.
