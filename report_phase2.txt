CHAPTER 1
INTRODUCTION

Cybersecurity in the digital era demands more than isolated detection tools. As organizations expand their digital footprints across cloud platforms, containerized workloads, and remote endpoints, the attack surface grows proportionally. Traditional security operations rely on siloed tools such as standalone antivirus software, periodic vulnerability scanners, and manual incident review processes. These approaches generate excessive alert noise, fail to correlate related events, and provide no structured memory of past incidents.

Phase 1 of the SentinelNexus project established the foundational architecture for an integrated cybersecurity platform. It implemented the initial vulnerability scanning pipeline using OSV Scanner, Trivy, and Semgrep, a host telemetry collection system, a basic ClamAV-based malware detection service, and a React-based dashboard with an Ollama-powered AI assistant. However, it lacked the deeper intelligence required for autonomous threat resolution.

Phase 2 significantly extends the platform by introducing critical additions that fulfill the complete proposed architecture: a real YARA-based static analysis engine for full file-level threat detection, a VirusTotal API integration for external hash reputation lookup, a SQLite-backed incident memory store for persistent threat tracking, and the ThreatSentinel correlation engine that aggregates findings across all scanners into unified, actionable threat events. Furthermore, Phase 2 adds a multi-factor Risk Scoring Engine and a PatchMaster remediation advisor that maps detected CVEs to executable fix commands. 

Together, these additions transform SentinelNexus from an isolated monitoring dashboard into a comprehensive, proactive, end-to-end threat detection, multi-source correlation, risk quantification, and guided remediation platform.


CHAPTER 2
LITERATURE SURVEY
2.1 Next Steps in Cyber Blue Team Automation — Leveraging the Power of LLMs (2025)
Authors: A. Dijk, R. Meier, C. Melella
This paper explores how Large Language Models (LLMs) and Generative AI can revolutionize blue team operations the defensive side of cybersecurity. Traditionally, blue teams depend heavily on manual work to review logs, triage alerts, write incident reports, and coordinate responses. These tasks are repetitive, time-consuming, and prone to fatigue and errors. The authors argue that the new generation of AI systems, particularly LLMs, can dramatically automate, assist, and enhance these processes.
Limitations of the paper:
1. The challenge of high-quality labelled datasets.
2. Integration complexity.
3. Prompt engineering and model explainability.
4. Computational resources and cost.

2.2 A Systematic Review of Cyber Threat Intelligence (2025)
Authors: P. Santos, R. Martins, F. Rodriguez, and A. Kumar
This paper is a systematic review of Cyber Threat Intelligence (CTI) — a field focused on collecting, analyzing, and sharing information about cyber threats to improve organizational defense. The review highlights that CTI is now a cornerstone of modern cybersecurity, used by governments, enterprises, and research organizations. It plays a vital role in detecting patterns of malicious activity, predicting attacker behavior, and preparing proactive defenses.
Limitations of the paper:
1. Many CTI studies are qualitative or case-study based.
2. Data sharing/trust remains a barrier.
3. Lack of uniform standards.

2.3 A Comprehensive Review and Assessment of Cybersecurity Vulnerability Detection Methodologies (2024)
Authors: K. Bennouk, M. El Yazid, A. El Omri
This paper provides a detailed overview of the different techniques and tools that researchers and organizations use to find security vulnerabilities in software systems. A vulnerability is basically a weakness in code, configuration, or design that an attacker can exploit to gain unauthorized access or cause damage. 
Limitations of the paper:
1. Many studies rely on curated datasets.
2. Vulnerability databases aren’t always accurately structured.
3. Static or dynamic methods often generate lots of false positives.

2.4 A Survey on Malware Detection and Analysis (2024)
Authors: Joshua Smallman
This paper offers a comprehensive review of established malware detection methods used to identify, analyze, and mitigate malicious software that threatens computer systems worldwide. It categorizes detection approaches into several foundational techniques, including signature-based detection, behavior-based detection, static analysis, dynamic analysis, and hybrid detection systems.
Limitations of the Paper:
1. Dependency on Known Patterns.
2. Static Analysis Weaknesses.
3. Dynamic Analysis Overheads.

2.5 YARA Rules in Malware Detection and Threat Intelligence (2024)
Authors: El-Bolock, A., Bousselham, R., Elgazzar, K.
This paper examines the practical effectiveness of YARA rules as a lightweight static analysis mechanism for identifying malware families. YARA (Yet Another Ridiculous Acronym) allows security researchers to define pattern-matching rules based on binary strings, regular expressions, and metadata conditions. The authors demonstrate that YARA rule scanning, when combined with hash-based reputation lookup, achieves significantly lower false-positive rates than antivirus-only approaches.
Limitations: YARA rules are only as effective as their authorship — unmaintained or generic rules produce high false-positive rates.

2.6 VirusTotal Integration for Enterprise Threat Intelligence Pipelines (2024)
Authors: Garcia, F., Hernandez, J., Uribe, C.
This paper proposes architectural patterns for integrating VirusTotal's public API into automated threat intelligence workflows. The authors discuss rate-limiting constraints of the free tier, caching strategies to avoid redundant lookups, and the value of multi-engine consensus.
Limitations: VirusTotal's free tier has strict rate limits. Unknown or newly compiled malware will not appear in the database.

2.7 Incident Correlation in Modern Security Operations Centers (2025)
Authors: Balogun, T., Schwartz, A., Lee, J.
This systematic review examines how correlation engines in modern SOC platforms reduce alert fatigue. The authors analyze twenty-three correlation approaches including rule-based, graph-based, and machine learning driven methods.
Limitations: Correlation engines trained on specific datasets may miss novel attack chains.

2.8 Risk Scoring Frameworks for Vulnerability Prioritization (2025)
Authors: Okonkwo, E., Fischer, R., Patel, S.
This paper surveys risk scoring methodologies used in enterprise vulnerability management programs. The authors compare CVSS-only scoring against multi-factor approaches that incorporate asset criticality, exploitability evidence (such as VirusTotal detections), and environmental context.
Limitations: Multi-factor scores require continuous data feeds and may be difficult to reproduce consistently across scan cycles.

2.9 Automated Remediation Guidance Using CVE-to-Package Mappings (2024)
Authors: Zhao, W., Kumar, A., Thompson, D.
This paper investigates the feasibility of automatically generating remediation commands from vulnerability scan outputs. The authors analyze popular package ecosystems (pip, npm, apt, gem, cargo) and demonstrate that CVE-to-package-manager mappings can be inferred from Trivy and OSV scanner outputs with high accuracy.
Limitations: Auto-generated commands may fail for packages with complex dependency graphs.


CHAPTER 3
EXISTING SYSTEM IMPERFECTIONS

3.1 Existing Baseline
Traditional cybersecurity environments rely on fragmented, independently operated tools. Security Information and Event Management (SIEM) systems aggregate logs but produce overwhelming alert volumes without structured correlation. Point-in-time vulnerability scanners report findings but do not track remediation history or recurring exposure. Antivirus tools such as ClamAV provide signature-based detection but lack static analysis depth and external reputation validation. Threat intelligence platforms provide indicator feeds but require manual analyst correlation to connect them to system-specific findings.

3.1.1 Operations of the Pre-Phase-2 System
1. Independent Tool Operation: Each scanner (antivirus, vulnerability scanner, SIEM) runs in isolation. Results are stored in separate formats and locations, requiring manual effort to understand compound risk.
2. No Persistent Incident Memory: Traditional systems treat each scan cycle as independent. A vulnerability detected in one run and left unpatched generates a new alert in the next run with no historical context about its age or prior investigation.
3. Manual Alert Correlation: Analysts must manually determine whether a ClamAV detection and a high-severity CVE in the same system are related. Without structured correlation, the connection is often missed.
4. Static Recommendations: Responses consist of generic advice. No tool automatically maps a specific CVE to the exact package manager command needed to patch it.
5. No External Hash Validation: Standard scanning does not query external intelligence databases. A file flagged cannot be automatically cross-referenced against the VirusTotal multi-engine consensus.

3.1.2 Limitations Addressed by Phase 2
1. Alert Fatigue: Unconnected per-tool alerts overwhelm analysts. Critical patterns hidden across tools go unnoticed.
2. No Memory of Past Incidents: Systems start fresh each cycle, missing recurring attacks and unresolved exposures.
3. Weak Malware Analysis Depth: ClamAV alone cannot detect files with obfuscated or novel payloads that match known behavioral patterns (YARA rules) but have modified signatures.
4. No Risk Quantification: There is no unified score telling operators which combination of findings represents the highest current risk.
5. Manual Remediation Lookup: Analysts must manually search for the correct patch command for each CVE, adding minutes per finding to total remediation time.


CHAPTER 4
PROPOSED SENTINELNEXUS PHASE 2 SYSTEM

The Phase 2 SentinelNexus platform extends the Phase 1 foundation by integrating the major modules that were missing from the original deployed architecture, turning a passive dashboard into an active, intelligent defense platform.

4.1 Objectives of Phase 2

1. Comprehensive Static Malware Analysis (YARA Engine)
Deploy a functioning YARA-based static analysis engine that scans files for known malicious patterns using community-maintained rule sets (Neo23x0/signature-base). The engine produces MD5, SHA-1, and SHA-256 multi-hash fingerprints for each scanned file, enabling external intelligence lookups.

2. External Hash Reputation Validation (VirusTotal Integration)
For every file flagged by ClamAV or YARA, submit its SHA-256 hash to the VirusTotal public API v3. Return the multi-engine detection count, malware family classification, and threat label.

3. Persistent Incident Memory (SQLite Store)
Maintain a structured database of all detected incidents, classified by severity, scanner source, and status (open or resolved). Track recurring attack patterns separately so repeated exposures are immediately visible. 

4. Threat Correlation Engine (ThreatSentinel)
After each combined scan cycle, aggregate findings from ClamAV, YARA, VirusTotal, and Trivy into unified ThreatEvent objects. Sort events by severity and persist them to a cache file accessible to the frontend in real time.

5. Multi-Factor Risk Scoring (Risk Scoring Engine)
Compute a 0-100 platform risk score by combining: ClamAV infection count, YARA rule hit count, VirusTotal detection positives, critical and high vulnerability count. Return a human-readable risk label (LOW/MEDIUM/HIGH/CRITICAL) with a breakdown of each contributing factor.

6. Actionable CVE-to-Remediation Mapping (PatchMaster)
Parse Trivy outputs to extract affected package names, installed versions, and package ecosystems. Generate the exact package manager command required to patch each vulnerability with a direct link to the CVE advisory.

4.2 Expanded System Architecture

Layer 1: The Unified Detection Engine
ClamAV (signature-based antivirus) now operates in parallel with the YARA Engine (pattern-based static analysis) and VirusTotal API (external reputation lookup). These three detection mechanisms together cover known signatures, behavioral patterns, and global intelligence consensus.

The SQLite Memory Module stores all incidents across scan cycles and tracks recurring patterns. The ThreatSentinel Correlator reads from all scanner outputs after each cycle and produces a unified threat event list.

The Risk Scoring Engine reads from all sources to compute the overall platform risk score. PatchMaster generates actionable fix commands, displayed to analysts in the Response page with one-click copy functionality. The Blue Team Copilot uses this correlated data to provide plain-language explanations.


CHAPTER 5
DESIGN AND IMPLEMENTATION DEEP DIVE

5.1 Design Blueprint

Phase 2 preserves the modular architecture established in Phase 1 and significantly enriches the Data Processing and Decision layers.
Design Goals:
- Complete the malware analysis pipeline: Cover known signatures, behavioral patterns, and global intelligence consensus.
- Build reliable cross-cycle memory: Empower the system to track incidents over time rather than treating each run independently.
- Provide honest risk quantification: Generate a single, actionable score based on actual detected findings.
- Enable analyst-executable remediation: Reduce the lookup step for patching CVEs to a single click.
- Eliminate static visual placeholders: Connect all frontend dashboard elements to live backend APIs.

5.2 Implementation Specifics

5.2.1 YARA Engine Implementation (backend/app/services/yara_engine/scanner.py)
The engine utilizes the `yara-python` and `gitpython` libraries. On first startup, it automatically clones the community Neo23x0/signature-base repository (containing thousands of rules) and compiles them alongside a hardcoded built-in fallback set. During scans, files are read into memory and hashed using hashlib (MD5, SHA-1, SHA-256). The compiled YARA rules are applied, and matches are returned as structured dictionaries containing the namespace, tags, metadata, and string match offsets.

5.2.2 VirusTotal Integration (backend/app/services/virustotal/lookup.py)
The module leverages the VirusTotal v3 public API. It accepts extracted SHA-256 hashes and handles HTTP 429 rate limiting gracefully by inserting delays for free-tier users. The response is normalized into a threat dict detailing total engine counts, positive detection counts, and specific malware classifications.

5.2.3 SQLite Memory Store (backend/app/services/memory/db.py)
A lightweight relational database is implemented using Python's built-in sqlite3. It maintains two primary tables: 'incidents' and 'patterns'. The `add_incident()` function tracks real-time threat detection, while `update_pattern()` watches for recurring threat vectors. The `resolve_incident()` function provides the analyst workflow for closing completed investigations.

5.2.4 ThreatSentinel Correlator (backend/app/services/threat_sentinel/correlator.py)
The correlator functions as a data pipeline that ingests `malware_status.json`, `latest.json` (Trivy), and active YARA/VT logs. It groups these into unified ThreatEvent objects, automatically escalating High/Critical severity items to the SQLite database, and sorts the output for real-time frontend consumption.

5.2.5 Multi-Factor Risk Scoring Engine (backend/app/services/risk_engine/scorer.py)
The Python-based scorer compiles a 0-100 metric by mapping specific weights to inputs:
- ClamAV positive hits (max 35)
- YARA pattern matches (max 25)
- VirusTotal multi-engine consensus (max 20)
- Trivy Critical/High vulnerability counts (max 15)
- Wazuh alerts (max 5)
It outputs a defined tier (LOW, MEDIUM, HIGH, CRITICAL) mapping dynamically directly to the frontend badge UI.

5.2.6 PatchMaster Remediation Advisor (backend/app/services/patchmaster/advisor.py)
This module parses the OSV/Trivy outputs and maps the ecosystem parameter against a dictionary of known package managers (npm, pip, apt). It returns a concatenated command string (e.g., `pip install --upgrade package==1.2.3`) alongside the exact CVE link. 

5.2.7 Malware Pipeline Orchestration (backend/auto_scanner.py)
The `run_malware_scan()` pipeline was rewritten to chain the agents seamlessly:
Stage 1: ClamAV Scan
Stage 2: YARA Engine Scan (Static Behavioral)
Stage 3: Hash Extraction
Stage 4: VirusTotal Engine Lookup
Stage 5: Result aggregation and JSON persistence

5.2.8 UI/UX Frontend Integration
The React SPA architecture was refactored to fetch exclusively from standard REST endpoints. The Detections, Response, Incidents, Malware, and Dashboard tabs now operate without any hardcoded data objects, representing a true real-time operational dashboard.


CHAPTER 6
RESULTS AND EVALUATION

6.1 Overview of Successful Module Deployment

The Phase 2 milestone has successfully brought SentinelNexus from a conceptual dashboard into a fully functioning, end-to-end security platform. The system now autonomously processes real files, correctly tags them using community YARA rules, looks up their global reputation, persists the incidents across system reboots, and generates the exact commands required to fix vulnerabilities.

6.2 Specific Capability Verifications

1. YARA Static Intelligence Verification
The YARA module was tested against known patterns. It correctly triggered the built-in rules, provided accurate byte offsets for the malicious strings, and successfully retrieved the community signature-base updates on startup.

2. VirusTotal Consensus Verification
When presented with known malicious SHA-256 hashes generated by the YARA engine, the VT integration successfully queried the API, respected the rate limits, and populated the database with threat classifications accurately imported from the public VT index.

3. Persistent Memory Verification
The SQLite implementation correctly persisted incident states. When a Vulnerability was triggered, it could be marked 'Resolved' via the frontend UI. Subsequent restarts of the scanner did not overwrite the resolved state, and repeated instances correctly incremented the 'hit_count' in the patterns table.

4. ThreatSentinel Correlation Validation
The ThreatEvents API correctly stitched together disparate scanner outputs. A single host exhibiting a ClamAV detection and a Critical CVE was clustered and properly elevated to the top of the Detections feed based on the correlation weight logic.

5. PatchMaster Output Validation
For detected Python vulnerabilities, the engine correctly formatted `pip` upgrade commands. For operating system level CVEs, it accurately generated `apt` upgrade commands, significantly demonstrating the real-world reduction in manual analyst effort required to secure a machine.

6.3 Dashboard and Copilot Validation
The Blue Team Copilot continues to successfully query the Ollama LLM. It was verified that the contextual injection now correctly feeds the LLM with the live SQLite metrics, the actual Vulnerability count, and the exact Malware scores, ensuring the Copilot's answers are strictly grounded in the reality of the machine's current state.


CHAPTER 7
CONCLUSION AND FUTURE SCOPE

7.1 Conclusion

The Phase 2 SentinelNexus platform successfully realizes the vision of a unified, AI-supported, end-to-end cybersecurity ecosystem. By engineering and successfully deploying the YARA static analysis engine, the VirusTotal integration, the SQLite persistent memory module, the ThreatSentinel correlator, and the automated PatchMaster tool, the system eliminates the traditional requirement for analysts to manually cross-reference isolated security dashboards.

The platform provides a comprehensive three-stage analytic pipeline against malware (Signature, Behavioral, and Reputation), quantifies risk objectively, tracks threats over time persistently, and actively tells the operator how to fix discovered flaws. Combined with the Blue Team Copilot translating and explaining these actions, SentinelNexus represents a significant step forward in making enterprise-grade security operations accessible, autonomous, and explainable.

7.2 Future Scope

1. Fully Automated Remediation (PatchMaster Active)
Currently, commands are presented for manual execution. Future updates could implement a governed 'Auto-Fix' mode that executes these commands via SSH or local agents on non-production systems automatically.

2. Enhanced Sandbox Detonation Integration
While Phase 2 includes robust static analysis, Phase 3 could integrate Cuckoo Sandbox natively into the backend orchestration to provide automated dynamic execution tracing for unknown files.

3. Neo4j Graph Database Upgrades
Migrating the SQLite relational memory to a full Graph Database (Neo4j) would allow for far deeper lateral movement mapping, connecting IPs, users, and CVEs into a multidimensional map.

4. Multilingual LLM Enhancement
Applying multi-language models to the Copilot system to support global SOC teams seamlessly without needing English proficiency.

5. Dynamic Red Team Probing
Engineering an autonomous module that continuously simulates phishing or lateral movement attacks to dynamically test and tune the scanners in real-time.
